# Repo Orientation: Eridian

> Generated by /orient:orient. Re-run to update.

## One-line purpose
Eridian is a TUI-based living dictionary builder that lets you speak words in an unknown language, assign English meanings, and translate back via nearest-neighbour centroid matching in embedding space — inspired by the alien communication process in Project Hail Mary.

## Primary language(s)
- Go (sole language — CLI, TUI, storage, and all planned audio/ML processing)

## Pipeline / workflow stages
1. **Config sync** — On startup, Go ensures `~/.eridian/` exists, scans for `.db` files, and reconciles `config.json` with what's on disk
2. **TUI interaction** — Bubble Tea renders a root menu; selecting an option delegates to a child screen model (new, list, label, translate, edit, status, clean)
3. **Language creation** — User types a name; Go creates a new per-language SQLite database with `Entry` and `Centroid` tables via GORM auto-migration
4. **Audio capture** — (planned) Go captures microphone input
5. **MFCC / power spectrum analysis** — (planned) Go computes audio features directly (no external Python service)
6. **Centroid computation via KNN** — (planned) Go computes centroid vectors and performs nearest-neighbour lookup for translation
7. **Storage** — Go writes entries (label + embedding) and centroids (label + averaged vector) into per-language SQLite databases

## Key files
- `main.go` — Go entry point; delegates to `cmd.Execute()` | Start here to trace the call chain
- `cmd/root.go` — TUI orchestration: Bubble Tea root model, screen routing, menu rendering, Cobra root command | The architectural backbone of the app

## Core concepts
**Centroid matching, not a classifier**: Instead of training a classifier, each English label's "voice signature" will be the average (centroid) of all its embedding vectors. Translation is nearest-neighbour lookup in embedding space. Adding a new word is just a new centroid — no retraining required.

**Go-owns-the-DB boundary**: Go is the single writer to SQLite. The Python ML service was recently purged (see commit `3b4d9db`), and audio/embedding processing is planned to move entirely into Go. This avoids cross-language IPC and concurrent write conflicts.

**Meaning units, not sentences**: Eridian explicitly does not produce grammatically correct sentences or do LLM-based reordering. It surfaces individual word-level meanings, which parallels how pidgin communication bootstraps between two parties with no shared language.

**Per-language SQLite files**: Each language is a self-contained `.db` file under `~/.eridian/` with two tables (`entries`, `centroids`). Creating a language creates a file; the active language is tracked in `config.json`. The config sync on startup ensures these stay consistent.

**Child screen pattern (Bubble Tea v2)**: The TUI uses a parent-child model where `rootModel` delegates to child models (e.g., `newModel`) based on the active screen enum. Only `cmd/new.go` is fully implemented — the rest are placeholder skeletons following the same interface.

## Common gotchas
1. **Bubble Tea v2 package paths** — The TUI uses `charm.land/bubbletea/v2` and `charm.land/bubbles/v2`, not the older `github.com/charmbracelet/bubbletea`. Mixing v1 and v2 imports causes type incompatibilities. Always use the `charm.land/` paths.

2. **Most screens are skeletons** — Only `cmd/new.go` has real logic (text input, DB creation, setting active language). The other screen files (`label.go`, `translate.go`, `edit.go`, `list.go`, `status.go`, `clean.go`) are placeholder stubs that just render a label and handle quit/escape. Don't assume they contain working functionality.

3. **Config sync must succeed before TUI starts** — `root.go` calls `config.SyncConfig()` before launching Bubble Tea. If `~/.eridian/` doesn't exist, sync creates it. If the config references a language whose `.db` was deleted, sync clears the active language. The TUI assumes sync has already run.
