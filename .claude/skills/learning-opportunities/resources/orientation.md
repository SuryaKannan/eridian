# Repo Orientation: Eridian

> Generated by /orient:orient. Re-run to update.

## One-line purpose
Eridian is a living dictionary builder that lets you speak words in an unknown language, assign English meanings, and translate back via nearest-neighbour embedding matching -- inspired by the alien communication process in Project Hail Mary.

## Primary language(s)
- Go (CLI, TUI, storage, orchestration)
- Python (ML service: audio-to-embedding via FastAPI)

## Pipeline / workflow stages
1. **Audio capture** -- Go records a spoken word via microphone
2. **Embedding generation** -- Go sends audio to the Python FastAPI service, which runs Wav2Vec 2.0 and returns an embedding vector
3. **Storage** -- Go writes the embedding + English label into a per-language SQLite database (`~/.eridian/<language>.db`)
4. **Centroid rebuild** -- Go recomputes centroid vectors (averaged embeddings per label) after any data mutation
5. **Translation** -- On new audio input, Go gets the embedding from Python, then finds the nearest centroid via cosine similarity and returns its label

## Key files
- `main.go` -- Go entry point; delegates to `cmd.Execute()` | Start here to trace the call chain
- `cmd/root.go` -- TUI orchestration: Bubble Tea root model, screen routing, menu rendering | The architectural backbone of the interactive app
- `cmd/new.go` -- "Create new language" screen with text input | Best example of a fully wired child screen pattern

- `docs/research.md` -- Linguistic research grounding the design (NSM, pidgin formation, attrition) | Essential context for why the tool works the way it does

## Core concepts
**Centroid matching**: Instead of training a classifier, each English label's "voice signature" is the average (centroid) of all its embedding vectors. Translation is nearest-neighbour lookup in embedding space. Lives in the `centroids` table of each language's `.db` file, rebuilt automatically after any mutation.

**Go-owns-the-DB boundary**: Go is the single writer to SQLite. Python never touches the database -- it receives audio, returns embeddings over HTTP, and is otherwise stateless. This avoids SQLite concurrent-write issues and lets most CLI commands work without Python running.

**Ephemeral port**: Go finds a free port at startup and passes it to the Python service via environment variable (`ERIDIAN_PORT`). No port configuration or collisions.

**Meaning units**: Eridian explicitly does not produce grammatically correct sentences or do LLM-based reordering. It surfaces individual word-level meanings, which parallels how pidgin communication bootstraps between two parties with no shared language.

**Per-language SQLite files**: Each language is a self-contained `.db` file under `~/.eridian/` with two tables (`entries`, `centroids`). Creating a language creates a file; deleting one removes it. The active language is tracked in `config.json`.

## Common gotchas
1. **Bubble Tea v2 vs v1 package paths** -- The TUI uses `charm.land/bubbletea/v2` and `charm.land/bubbles/v2`, not the older `github.com/charmbracelet/bubbletea`. Mixing v1 and v2 imports causes type incompatibilities (e.g., `Cmd` and `Msg` types don't match across versions). Always use the `charm.land/` paths.

2. **AltScreen is per-View in v2** -- Unlike Bubble Tea v1 where `tea.WithAltScreen()` is a program option, v2 controls it via `view.AltScreen = true` on the `tea.View` return value. Only the root `View()` needs to set this since child views are rendered through it.

3. **Child model Init() must be called manually** -- When switching screens in `root.go`, the child model's `Init()` must be returned as a `tea.Cmd` (e.g., `return m, m.childModel.Init()`), otherwise components like text input cursors won't animate.
